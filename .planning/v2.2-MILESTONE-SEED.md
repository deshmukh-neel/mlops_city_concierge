## Working name

**v2.2 — Reasoning-Model Decisiveness** (the behavioral half of what v2.1's plumbing
enabled).

## Trigger / problem statement

v2.1 shipped per-provider reasoning-state preservation (Phase 9, PR #103): provider
reasoning state round-trips byte-correctly through `graph.ainvoke` for all four families
(gpt-5, DeepSeek reasoner, Claude, Gemini 3), zero wire errors. **The plumbing works.
The models still don't commit.** Every reasoning model burns its step budget exploring
and never (or rarely) calls `commit_itinerary`:

| Model | committed_itinerary_rate | Evidence (clean matrix `eval_reports/2026-06-05T20-29-56Z/`) |
|---|---|---|
| openai/gpt-4o-mini (v2.0 anchor) | **1.0** (5/5) | commits in 49-81s, 12-18 tool calls |
| openai/gpt-5-mini | **0.4** (2/5) | 3/5 step-limit at 153-188s; milestone anchor gate (≥0.6) FAILED |
| deepseek/deepseek-reasoner | **0.0** (0/5) | all step-limit, never calls commit_itinerary, 103-152s |
| anthropic/claude-sonnet-4-6 | 1.0 (n=1 probe only) | n=5 deferred to Phase 11 BASE-01 |
| gemini/gemini-3.1-pro-preview | unmeasured | first n=5 in Phase 11 BASE-01 |

This is a **behavioral/architectural gap, not an adapter gap** — tool/prompt micro-tweaks
were already tried and failed on DeepSeek (memory `project_deepseek_decisiveness_gap`).

## Load-bearing findings from the 2026-06-10 analysis (do not re-derive)

1. **The critique loop is NOT the main driver for gpt-5-mini.** Its 3 clean-run
   step-limit runs had `revision_reasons = []` — zero critique hints fired. The model
   burned 8 tool-calling steps exploring without committing. For deepseek-reasoner,
   `low_similarity` hints fired in 3/5 runs (secondary pressure, not root cause).
2. **`LOW_SIMILARITY_THRESHOLD` already IS 0.55** (`app/agent/revision.py:21`, unchanged
   since the file was extracted). The "deferred fix: set it to 0.55" recorded in
   09-04-SUMMARY is a no-op as written — if threshold tuning is attempted, the change is
   *lowering it below* 0.55, and it must be co-tuned with commit instructions (memory
   `critique-loop-and-commit-tool-conflict`: tightening one regresses the other).
3. **Finalize-on-commit works; critique cannot block a successful commit.** A commit
   routes to `critique_final_with_stops` (`app/agent/graph.py:443-457`), which runs only
   `itinerary_violations` + vibe — never the low-similarity diagnosis. Models that fail
   do so by *never calling* `commit_itinerary`, not by being blocked after calling it.
4. **Hypothesized mechanism (from `project_reasoning_models_break_agent_loop`):**
   SYSTEM_PROMPT rule 8's commit precondition is "ONE VIABLE OPTION for each requested
   stop" — reasoning models apply their own stricter notion of "viable" to
   sub-0.55-cosine results, so the precondition never feels satisfied. They also read
   rubric-shaped instructions as descriptions-to-reason-about rather than orders
   (gpt-5.4-mini once committed 1 stop of 3 because "the user is editing one slot" —
   a defensible reading the rubric punishes).
5. **Latency is call-count × per-call reasoning cost, not overhead.** One LLM call per
   plan step; critique adds zero LLM calls; tools execute sequentially
   (`graph.py:340-402`). gpt-5-mini ≈ 90s/turn even on *successful* runs. Step-count
   reduction (= decisiveness) is the dominant latency lever.
6. **Residual state-preservation gaps (carried from the analysis, bundle as experiment
   arms, not standalone fixes):**
   - Replay is single-message: only the most recent AIMessage's `_reasoning_state` is
     replayed (`graph.py:307-312`); earlier in-window tool-calling AIMessages get none.
   - `_prune_for_llm` collapses pre-cutoff AIMessage list-content via `str()`
     (`graph.py:227`) — destroys content-block reasoning; only `additional_kwargs`
     survive (D-08-07).
   - Reasoning state is intra-request only; `/chat` rebuilds history text-only
     (`app/agent/io.py:32-40`). Likely correct by design, but verify.

## Draft scope (phases, falsifier-shaped)

Design as joint experiments over three coupled levers — prompt contract, critique
pressure, state richness — never one lever in isolation:

- **Phase v2.2-1 — Decisiveness instrumentation + falsifier.** Add per-run telemetry
  (steps-to-first-commit-consideration, viable-candidate counts at each step, whether
  rule 8's precondition was objectively met when the model kept searching). Define the
  falsifier: an intervention "works" iff gpt-5-mini commit rate moves ≥0.6 at n=5 with
  gpt-4o-mini ≥ its honest baseline (no anchor regression).
- **Phase v2.2-2 — Viability contract / forced-commit experiments.** Candidate arms:
  (a) explicit viability definition in the commit precondition ("a result above X cosine
  with matching primary_type IS viable — do not keep searching past it");
  (b) forced-commit-at-step-N mechanism (graph-level, model-independent);
  (c) critique recalibration co-tuned with (a) (threshold below 0.55 and/or scoping
  low_similarity to pre-candidate steps only);
  (d) parallel tool execution in `act()` (latency arm, benefits all providers).
  Judge arms with the Phase-10 honest gates, n=5, temp=1.0.
- **Phase v2.2-3 — Richer state replay (only if v2.2-2 plateaus).** Multi-message
  replay; preserve content blocks through prune. Justify with an A/B, not by assumption.
- **Phase v2.2-4 — Cross-model gate promotion + baseline regen.** Whatever arm wins,
  regenerate baselines (Phase-10/11 infrastructure makes this cheap and honest) and
  promote reasoning-model gates from logged-not-gated to enforced where earned.
