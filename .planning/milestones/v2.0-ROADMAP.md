## Overview

Move the City Concierge agent from demo-ready to hostable. Build the eval infrastructure
that lets every subsequent fix be scored against committed baselines across multiple
providers, then fix the three agent-behavior bugs that survive the PR #94 reliability
merge: category compliance, rationale-stop alignment, and minimal-edit refinement.

All 21 v2.0 requirements (OVR / EVAL / CAT / RAT / REF) mapped 1:1 to phases. Five
phases shipped as feature branches PR'd into `main`; final PR (#100) brought Phase 6
to origin/main on 2026-06-03.

## Milestone Summary

**Decimal Phases:** None — milestone scope held without insertion phases.

**Key Decisions:**

- Eval harness ships BEFORE agent-behavior fixes in v2.0 (decided 2026-05-21). Without it, fix evaluation is one-run-at-a-time eyeballing — the failure mode that surfaced this milestone.
- Model selection for testing uses an env-var override, not MLflow alias surgery (decided 2026-05-21). Shared `production` alias has caused incidents (Gemini 3 rollback); env-var keeps prod safe.
- PR #94 merges before v2.0 Phase 2 starts; re-baseline confirms which bugs are real (decided 2026-05-21). Branch-vs-main confusion during initial v2.0 scoping mis-attributed which bugs existed where.
- Drop "step-budget tuning" from v2.0 scope (originally 6 phases → 5 phases) (decided 2026-05-21). Post-merge re-baseline showed the original "step-limit" observation was the JSON-safety crash misattributed.
- D-04-09: RAT-01 + RAT-03 fold into Phase 4 (structurally same problem as CAT-02); Phase 5 narrows to RAT-02 only.
- D-04-13 / D-04-14: When the scorer being gated is added in the same phase, substitute an absolute floor for the undefined delta-vs-old-baseline; document the trade-off as a locked decision.
- Phase 6 D-06-09 part 2 accepted with notes (decided 2026-06-03): pre-Phase-6 1.0 baselines were Phase-4 fail-open false positives; the new fail-loud measurement makes the "regression" appear, but real agent behavior improved. Real fix (reasoning-content threading) scoped to v2.1.
- v2.1 milestone scope drafted (decided 2026-06-03): reasoning-state preservation in `_prune_for_llm`, prompt/rubric decoupling, Anthropic provider wiring, honest cross-model baseline regen.

**Issues Resolved:**

- Tool-call args staying JSON-serializable across plan steps (already shipped in PR #94, but Phase 3 EVAL-08 hard-pins it in CI).
- Category-compliance regressions (P7 mitigation via schema enforcement of `primary_type_family`).
- Closure-swap placeholder bleed into user-facing rationale text (RAT-02 via inherit-or-synthesize at construction site).
- Refinement turns re-shuffling unchanged stops (REF-01 via structured `current_plan` + same-`primary_type` constraint).
- Stale-baseline drift between agent code and committed baselines (CI hard gate via `check_baselines_fresh.py`).
- Closure-detection sanity confirmed on 4 flagged Mission places before committing closure-related baselines (D-07 PASS).

**Issues Deferred (to v2.1):**

- Reasoning-content thread-through in `_prune_for_llm` (gpt-5-mini / gpt-5.4-mini / DeepSeek reasoner / Claude / Gemini 3 all currently fail on tool-loop tasks because reasoning state is dropped across turns).
- Prompt/rubric decoupling (move behavioral rules from prompt → scorer to enable honest cross-model comparison).
- Anthropic provider wiring (add `claude` to `SUPPORTED_PROVIDERS`).
- Honest cross-model baseline regen after reasoning-state fix.
- NOJ-01: N-of-M pass rate (`--runs-per-case N`) for temperature=1.0 model variance.
- JUD-01: LLM-as-judge scorer for rationale quality.
- OVR-07 / OVR-08: Per-PR ephemeral MLflow aliases, `/reload-model` admin endpoint.

**Technical Debt Incurred:**

- D-06-09 part 2 baseline regen captures a fail-open → fail-loud measurement shift, not a regression — but the reported scores look like a regression in isolation. Fix is the v2.1 reasoning-content work; no remediation possible within v2.0 architecture.
- Reasoning models cannot be used as agent drivers on this codebase. Locked `openai/gpt-4o-mini` as v2.0 prod anchor; revisit when v2.1 ships.
- CLO-01: Over-aggressive closure detection on Mission queries flagged for investigation. Deferred — could be hours-data drift, `place_is_open` timezone bug, or `_per_stop_closure_status` overly cautious.
