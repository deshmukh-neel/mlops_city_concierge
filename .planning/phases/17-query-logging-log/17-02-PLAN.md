<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `/chat` user → `log_user_query` → user_query_log | The raw user `message` (untrusted) crosses into a SQL write here. This is the primary boundary for this plan. |
| `/chat` request path → BackgroundTask threadpool | The log runs out-of-band after the response; a failure in the threadpool must not propagate to the request. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-04 | Tampering | SQL injection via raw `message` at write time | mitigate | The INSERT uses psycopg2 `%s` bound parameters; `message` is passed as a param, never string-formatted into SQL. Enforced by Task 1 param-safety unit test (asserts the message literal never appears in the executed SQL string). |
| T-17-05 | Denial of service | a DB failure in logging blocking/breaking `/chat` | mitigate | D-01 fire-and-forget via BackgroundTasks (runs after the response) + D-04 swallow-all-exceptions (`except Exception` + logger.warning, never re-raise). A logging failure cannot add latency to or break the reply. Verified by the fail-open unit tests (get_conn raising AND cur.execute raising). |
| T-17-06 | Information disclosure | PII in raw stored `message` | accept | D-04: raw verbatim store is an explicitly accepted risk (private capstone DB, low-PII SF place queries). Documented in the 17-01 migration comment. Not a blocker. |
| T-17-07 | Repudiation / Info disclosure | over-capture (full reply, committed place_ids) | mitigate | D-02 scope: only message + extracted slots + lightweight metadata are logged; full reply / place_ids are intentionally NOT captured (overlaps MLflow trace_request). No field added to ChatRequest/ChatResponse. |
| T-17-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed (fastapi `BackgroundTasks`, psycopg2, pytest-mock all already in pyproject.toml). No package-legitimacy checkpoint required. |
</threat_model>
